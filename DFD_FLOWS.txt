================================================================================
CORE ENTITIES
================================================================================

ENTITY: User
Primary Key: user_id (UUID)
Attributes:
- user_id: UUID (PK)
- email: String (UNIQUE)
- phone: String (UNIQUE)
- password_hash: String
- first_name: String
- last_name: String
- address: String
- city: String
- role: Enum (CITIZEN, ADMIN, OFFICER)
- status: Enum (ACTIVE, INACTIVE, DELETED)
- created_at: Timestamp
- updated_at: Timestamp

---

ENTITY: Department
Primary Key: department_id (UUID)
Attributes:
- department_id: UUID (PK)
- name: String (UNIQUE)
- email: String
- phone: String
- address: String
- city: String
- status: Enum (ACTIVE, INACTIVE)
- created_at: Timestamp
- updated_at: Timestamp

---

ENTITY: Complaint
Primary Key: complaint_id (UUID)
Attributes:
- complaint_id: UUID (PK)
- citizen_id: UUID (FK to User)
- department_id: UUID (FK to Department, NULLABLE)
- assigned_officer_id: UUID (FK to User, NULLABLE)
- title: String
- description: Text
- category: String
- latitude: Float
- longitude: Float
- address: String
- priority: Enum (LOW, MEDIUM, HIGH, URGENT)
- status: Enum (OPEN, IN_PROGRESS, RESOLVED, CLOSED, REJECTED, ESCALATED)
- resolution_details: Text
- created_at: Timestamp
- updated_at: Timestamp
- resolved_at: Timestamp (NULLABLE)

Foreign Keys:
- citizen_id → User(user_id)
- department_id → Department(department_id)
- assigned_officer_id → User(user_id) [where role=OFFICER]

---

ENTITY: ComplaintHistory
Primary Key: history_id (UUID)
Attributes:
- history_id: UUID (PK)
- complaint_id: UUID (FK to Complaint)
- previous_status: String
- new_status: String
- changed_by: UUID (FK to User)
- reason: Text
- created_at: Timestamp

Foreign Keys:
- complaint_id → Complaint(complaint_id)
- changed_by → User(user_id)

---

ENTITY: Feedback
Primary Key: feedback_id (UUID)
Attributes:
- feedback_id: UUID (PK)
- complaint_id: UUID (FK to Complaint)
- user_id: UUID (FK to User)
- rating: Integer (1-5)
- comment: Text
- created_at: Timestamp

Foreign Keys:
- complaint_id → Complaint(complaint_id)
- user_id → User(user_id)

---

ENTITY: Notification
Primary Key: notification_id (UUID)
Attributes:
- notification_id: UUID (PK)
- user_id: UUID (FK to User)
- complaint_id: UUID (FK to Complaint, NULLABLE)
- type: Enum (STATUS_UPDATE, ASSIGNMENT, ESCALATION, WELCOME)
- title: String
- message: Text
- channel: Enum (EMAIL, SMS, PUSH)
- status: Enum (PENDING, SENT, DELIVERED, READ)
- created_at: Timestamp
- sent_at: Timestamp (NULLABLE)

Foreign Keys:
- user_id → User(user_id)
- complaint_id → Complaint(complaint_id)

================================================================================
RELATIONSHIPS & CARDINALITY
================================================================================

1. User (Citizen) ↔ Complaint: ONE-TO-MANY
   Cardinality: (1) → (*)
   A citizen creates many complaints

2. Department ↔ Complaint: ONE-TO-MANY
   Cardinality: (1) → (*)
   A department handles many complaints

3. User (Officer) ↔ Complaint: ONE-TO-MANY
   Cardinality: (1) → (*)
   An officer handles many complaints

4. Complaint ↔ ComplaintHistory: ONE-TO-MANY
   Cardinality: (1) → (*)
   Each complaint has multiple status updates

5. Complaint ↔ Feedback: ONE-TO-MANY
   Cardinality: (1) → (*)
   Each complaint receives feedback

6. User ↔ Feedback: ONE-TO-MANY
   Cardinality: (1) → (*)
   Users provide feedback

7. User ↔ Notification: ONE-TO-MANY
   Cardinality: (1) → (*)
   Users receive multiple notifications

8. Complaint ↔ Notification: ONE-TO-MANY
   Cardinality: (1) → (*)
   Complaints trigger notifications

================================================================================
SIMPLIFIED ENTITY DIAGRAM STRUCTURE
================================================================================

User (Base Entity)
├── Roles: CITIZEN, ADMIN, OFFICER
├── Relations:
│   ├── Creates: Complaint
│   ├── Handles: Complaint (if OFFICER)
│   ├── Receives: Notification
│   ├── Provides: Feedback
│   └── Updates: ComplaintHistory

Department (Base Entity)
├── Relations:
│   └── Receives: Complaint
│   └── Managed by: Admin User

Complaint (Transaction Entity)
├── Relations:
│   ├── Created by: User (CITIZEN)
│   ├── Assigned to: Department
│   ├── Handled by: User (OFFICER)
│   ├── Has: ComplaintHistory (Status Changes)
│   ├── Receives: Feedback
│   └── Triggers: Notification

ComplaintHistory (Audit Trail)
├── Relations:
│   ├── Belongs to: Complaint
│   └── Updated by: User

Feedback (Rating Entity)
├── Relations:
│   ├── For: Complaint
│   └── From: User (CITIZEN)

Notification (Communication Entity)
├── Relations:
│   ├── To: User
│   └── About: Complaint

================================================================================
KEY PRINCIPLES
================================================================================

1. User Entity covers all roles (CITIZEN, ADMIN, OFFICER)
   - No separate Officer table needed
   - Role-based filtering in queries

2. Complaint is the central transaction
   - Links Users (Citizen + Officer), Department
   - Status tracked in ComplaintHistory
   - Notifications triggered from Complaint events

3. ComplaintHistory replaces Escalation table
   - ESCALATED is a status in Complaint
   - Status transitions logged in history

4. Minimal Foreign Keys
   - Reduced complexity
   - Easier to visualize and implement

5. Core Flows Remain:
   - Citizen creates Complaint
   - Admin assigns to Department
   - Officer handles & updates status
   - Notifications sent on changes
   - Feedback collected on resolution
